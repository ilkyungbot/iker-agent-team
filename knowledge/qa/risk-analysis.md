# 리스크 분석 — 테스트할 곳을 잘못 고르면 노력이 낭비된다

> 모든 코드를 완벽히 테스트할 수 없다. 위험한 곳을 먼저 테스트하라.

## 핵심 원칙
- 리스크 = 발생 확률 × 비즈니스 임팩트
- 테스트 투자는 리스크에 비례해야 한다
- 변경 빈도와 복잡도가 높은 모듈이 우선순위
- 외부 의존성(API, DB, 결제)은 항상 높은 리스크로 분류

## 판단 기준
| 리스크 등급 | 조건 | 테스트 수준 |
|------------|------|------------|
| Critical | 결제, 인증, 데이터 손실 | 단위+통합+E2E |
| High | 핵심 비즈니스 로직 | 단위+통합 |
| Medium | UI 컴포넌트, 유틸 | 단위 |
| Low | 설정, 타입 정의 | 정적 분석만 |

**리스크 식별 체크리스트:**
- [ ] 장애 시 매출에 직접 영향을 주는가?
- [ ] 복구가 어렵거나 데이터 손실을 유발하는가?
- [ ] 외부 시스템과 연동되는가?
- [ ] 여러 팀이 공유하는 코드인가?
- [ ] 최근 6개월 내 버그가 발생한 이력이 있는가?

## 코드 예시 (TypeScript/Vitest)
```typescript
// 결제 로직 — Critical 등급: 모든 엣지케이스 커버
describe('PaymentProcessor', () => {
  describe('결제 금액 계산', () => {
    it('할인 쿠폰 적용 시 최소 결제금액 1원을 보장한다', () => {
      const result = calculatePayment({ amount: 100, discount: 200 })
      expect(result.finalAmount).toBeGreaterThanOrEqual(1)
    })

    it('VAT 포함 금액이 올바르게 계산된다', () => {
      const result = calculatePayment({ amount: 10000, vatRate: 0.1 })
      expect(result.totalAmount).toBe(11000)
    })

    it('음수 금액 입력 시 에러를 던진다', () => {
      expect(() => calculatePayment({ amount: -100 })).toThrow(
        '결제 금액은 0보다 커야 합니다'
      )
    })
  })
})
```

## 안티패턴
- 리스크 평가 없이 작성하기 쉬운 테스트부터 시작
- Critical 기능에 Happy Path만 테스트
- 리스크 문서 없이 구두로만 공유
- 레거시 코드는 리스크가 낮다고 가정 (오히려 높다)

## 실전 팁
- 릴리즈 전 리스크 매트릭스를 팀과 30분간 리뷰한다
- 프로덕션 인시던트 이력을 리스크 평가에 반드시 반영한다
- 새 기능 추가 시 "이 코드가 망가지면 어떤 일이 생기는가?" 먼저 질문한다
- 리스크 등급은 분기마다 재평가한다
