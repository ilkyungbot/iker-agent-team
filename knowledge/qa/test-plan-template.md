# 테스트 플랜 템플릿 — 체계적인 테스트 준비를 위한 가이드

> 테스트 플랜은 계획이지 의식(ritual)이 아니다. 실용적으로 작성하라.

## 핵심 원칙
- 테스트 플랜은 짧고 실행 가능해야 한다 (긴 문서는 읽히지 않는다)
- 범위(Scope)와 비범위(Out of Scope)를 명확히 한다
- 리스크를 먼저 식별하고 테스트 우선순위에 반영한다
- 완료 기준(Exit Criteria)을 정량적으로 정의한다

## 판단 기준
테스트 플랜 필요 상황:
- 대규모 신규 기능 (2주 이상 개발)
- 크리티컬 시스템 변경 (결제, 인증, 데이터 마이그레이션)
- 외부 팀과 협업이 필요한 릴리즈
- 규정 준수(Compliance) 요건이 있는 기능

테스트 플랜 생략 가능 상황:
- 소규모 버그 수정
- 텍스트/UI 마이너 변경
- 기존 기능의 단순 확장

## 코드 예시 (테스트 플랜 마크다운 템플릿)
```markdown
# 테스트 플랜: [기능명]
- 버전: 1.0
- 작성자: @담당자
- 작성일: YYYY-MM-DD
- 리뷰어: @리뷰어

## 1. 범위
**테스트 대상:**
- 구독 플랜 업그레이드/다운그레이드 API
- 결제 처리 통합 (Stripe)
- 이메일 알림 발송

**테스트 제외:**
- Stripe 결제 게이트웨이 내부 동작
- 기존 구독 조회/목록 API (변경 없음)

## 2. 리스크 평가
| 리스크 | 영향도 | 발생 가능성 | 대응 |
|--------|--------|------------|------|
| 결제 실패 시 구독 상태 불일치 | 높음 | 중간 | 결제-구독 상태 동기화 테스트 집중 |
| 동시 다중 업그레이드 | 높음 | 낮음 | 멱등성 테스트 |

## 3. 테스트 전략
- 단위 테스트: 플랜 변경 계산 로직
- 통합 테스트: 결제 API 연동 (Stripe 샌드박스)
- E2E: 업그레이드 완료 시나리오

## 4. 완료 기준
- [ ] 커버리지 ≥ 85%
- [ ] 모든 Critical 테스트 통과
- [ ] 결제 실패 시나리오 3가지 이상 검증
- [ ] 성능 P95 < 500ms

## 5. 일정
| 단계 | 시작 | 완료 |
|------|------|------|
| 테스트 케이스 작성 | 3/20 | 3/22 |
| 자동화 구현 | 3/22 | 3/25 |
| 탐색적 테스트 | 3/25 | 3/26 |
| 릴리즈 승인 | 3/26 | - |
```

## 안티패턴
- 100페이지 테스트 플랜 작성 후 릴리즈 전날 확인
- 개발 완료 후 테스트 플랜 작성 (이미 늦음)
- 완료 기준 없이 "충분히 테스트했다" 판단
- 테스트 플랜을 QA만 작성하고 개발팀 미참여

## 실전 팁
- 기능 설계 단계에서 개발자와 QA가 함께 테스트 플랜을 작성한다
- 1페이지 이내로 작성하는 것을 목표로 한다
- Linear/Jira 이슈에 테스트 플랜을 첨부해 추적성을 확보한다
- 릴리즈 후 "테스트 플랜 회고"로 누락된 케이스를 분석한다
