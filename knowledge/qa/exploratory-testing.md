# 탐색적 테스트 — 스크립트 없이 제품을 탐험한다

> 자동화 테스트는 우리가 예상한 것만 잡는다. 탐색적 테스트는 예상 밖을 잡는다.

## 핵심 원칙
- 탐색적 테스트는 무작위가 아니다. 가설을 세우고 검증한다
- 시간 박스(60-90분)로 집중도를 유지한다
- 발견한 내용은 실시간으로 기록한다
- 자동화를 보완하는 것이지 대체하는 것이 아니다

## 판단 기준
탐색적 테스트가 효과적인 상황:
- 신규 기능 릴리즈 직전
- 대규모 리팩터링 후
- 자동화 커버리지가 낮은 영역
- 사용자 불만이 집중된 기능
- 복잡한 사용자 여정

탐색적 테스트 헤리스틱 (SFDIPOT):
- **Structure**: 데이터 구조의 경계값
- **Function**: 기능의 변형과 조합
- **Data**: 예상치 못한 입력값
- **Interface**: UI와 API 경계
- **Platform**: 다양한 환경/브라우저
- **Operations**: 운영 시나리오
- **Time**: 타이밍 관련 동작

## 코드 예시 (테스트 세션 노트 템플릿)
```markdown
## 탐색적 테스트 세션 #42
- 날짜: 2026-03-20
- 테스터: @tester
- 대상: 구독 업그레이드 플로우
- 시간: 60분
- 차터(목표): "프리 → 프로 업그레이드 시 엣지케이스 탐색"

### 탐색 가설
1. 업그레이드 중 결제 실패하면 어떻게 되는가?
2. 이미 다른 탭에서 업그레이드 중일 때 또 시도하면?
3. 업그레이드 완료 직후 탭을 닫으면 상태가 어떻게 되는가?

### 발견사항
- [BUG] 결제 실패 후 재시도 시 이전 입력값이 사라짐
  - 재현: 카드 번호 잘못 입력 → 에러 → 페이지 새로고침됨
  - 심각도: Medium
- [UX] 업그레이드 완료 메시지 후 3초간 버튼이 비활성화되지 않아
  중복 클릭 가능
  - 심각도: Low
- [INFO] 동시 업그레이드 시도는 서버에서 멱등성 처리됨 (정상 동작)

### 자동화 제안
- 결제 실패 후 폼 상태 유지 회귀 테스트 추가
- 버튼 중복 클릭 방지 E2E 테스트 추가
```

## 안티패턴
- 방향 없이 무작위로 클릭
- 세션 결과를 기록하지 않아 발견사항 소실
- 탐색적 테스트를 자동화의 대안으로 사용
- 발견한 버그를 당장 수정하려다 세션 중단

## 실전 팁
- 세션 전 탐색 차터(charter)를 30초 이내로 명확히 정의한다
- 타이머 앱으로 시간 박스를 엄격히 관리한다
- 버그 발견 시 재현 단계만 기록하고 세션 후 등록한다
- 페어 탐색(두 명이 함께)은 사각지대를 줄인다
