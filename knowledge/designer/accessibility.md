# 접근성 — 모든 사용자를 위한 디자인

> 접근성은 장애인을 위한 배려가 아니다. 더 나은 디자인이 모두를 이롭게 한다.

## 핵심 원칙

- **포용적 설계**: 다양한 능력과 맥락의 사용자를 처음부터 고려한다
- **WCAG 기준**: 최소 AA 등급 준수 (대비, 대체 텍스트, 키보드 탐색 등)
- **다중 감각 표현**: 색상만으로 정보를 전달하지 않는다 (아이콘, 텍스트 병용)
- **키보드 접근성**: 마우스 없이 모든 기능에 접근 가능해야 한다
- **스크린 리더 지원**: 의미 있는 HTML 구조 + aria 레이블

## 판단 기준

| 접근성 항목 | 기준 | 테스트 방법 |
|-----------|------|-----------|
| 텍스트 대비 | 4.5:1 (일반), 3:1 (대형) | Contrast Checker |
| 포커스 표시 | 명확한 focus ring | Tab 키로 탐색 |
| 이미지 대체 텍스트 | 의미 있는 alt 텍스트 | 스크린 리더 |
| 터치 타겟 | 최소 44×44px | 실기기 테스트 |
| 움직이는 콘텐츠 | 정지·일시정지 가능 | prefers-reduced-motion |
| 폼 레이블 | 명확한 label 연결 | 스크린 리더 |

## 실전 예시

**접근성 체크리스트 (디자인 완료 전)**
- [ ] 모든 이미지에 alt 텍스트 정의 완료
- [ ] 색상 대비비 확인 (Stark, Able 플러그인)
- [ ] 색맹 시뮬레이션 통과
- [ ] 키보드 포커스 순서 논리적
- [ ] 에러 메시지가 색상 외 텍스트로 표현됨
- [ ] 폼 필드에 placeholder 외 visible label 존재
- [ ] 모달/팝업의 포커스 트랩 설계 완료

**아이콘 버튼 접근성**
- 아이콘만 있는 버튼: aria-label로 목적 설명
- 장식용 아이콘: aria-hidden="true"
- 상태 변화 아이콘: aria-live로 실시간 알림

## 안티패턴

- "장애인이 우리 서비스를 쓰겠어?" → 접근성 = 일시적 제약 포함 (손 부상, 밝은 햇빛)
- placeholder를 label 대신 사용 → 입력 시 사라져 맥락 손실
- 색상으로만 오류 표시 → 적록색맹 사용자 배제
- focus outline을 `outline: none`으로 제거
- 텍스트를 이미지로 대체 (확대 시 깨짐, 스크린 리더 불가)

## 실전 팁

- Figma Stark 플러그인으로 디자인 단계에서 대비 체크
- Chrome DevTools → Accessibility 탭에서 DOM 트리 확인
- VoiceOver(macOS)/TalkBack(Android) 직접 테스트 주 1회 권장
- 접근성 수정은 개발 후보다 디자인 단계에서 하는 것이 10배 저렴
- WCAG 2.2 체크리스트를 팀 공유 노션에 붙여 두고 체크박스로 관리

## 심화: 접근성 구현 가이드

### WCAG 2.2 주요 성공 기준 (AA 등급)
| 기준 | 설명 |
|------|------|
| 1.1.1 | 모든 비텍스트 콘텐츠에 대체 텍스트 |
| 1.3.1 | 정보와 관계가 프로그래밍적으로 전달 |
| 1.4.3 | 텍스트 대비 4.5:1 이상 |
| 2.1.1 | 모든 기능을 키보드로 접근 가능 |
| 2.4.3 | 포커스 순서가 논리적 |
| 2.4.7 | 키보드 포커스가 시각적으로 보임 |
| 3.1.1 | 페이지 언어가 프로그래밍적으로 명시 |
| 4.1.2 | 모든 UI 컴포넌트에 이름/역할/값 |

### 스크린 리더 테스트 체크
**macOS VoiceOver 시작**: Cmd + F5
- 탭 이동으로 모든 인터랙티브 요소 접근 가능한지
- 아이콘 버튼의 목적이 읽히는지
- 폼 레이블이 올바르게 연결되어 있는지
- 오류 메시지가 포커스를 받거나 aria-live로 공지되는지

### 포용적 설계의 실제 수혜자
- 시각 장애: 대체 텍스트, 스크린 리더, 고대비
- 청각 장애: 자막, 텍스트 대안
- 운동 장애: 키보드 접근성, 큰 타겟
- 인지 장애: 명확한 언어, 일관성, 오류 예방
- 일시적 제약: 한 손 사용, 강한 햇빛, 시끄러운 환경
- 상황적 제약: 저속 인터넷, 오래된 기기

### 자동화 테스트 도구
- **axe DevTools**: Chrome 확장, 자동 접근성 검사
- **WAVE**: 웹 접근성 시각화 도구
- **Lighthouse**: Chrome DevTools 내장 접근성 점수
- **Figma Stark**: 디자인 단계 대비 체크

### 접근성 로드맵 우선순위
1. 이미지 alt 텍스트 (빠른 승리)
2. 색상 대비 수정
3. 키보드 포커스 순서 정의
4. 폼 레이블 연결
5. 아이콘 버튼 aria-label
6. 스크린 리더 전체 플로우 테스트
