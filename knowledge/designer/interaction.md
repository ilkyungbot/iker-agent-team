# 인터랙션 디자인 — 사용자 행동에 반응하는 UI 설계

> 인터랙션은 사용자와 시스템 사이의 대화다. 대화가 자연스러울수록 제품이 좋아진다.

## 핵심 원칙

- **피드백**: 모든 사용자 액션에 즉각적인 시스템 반응이 있어야 한다
- **어포던스**: UI 요소의 생김새가 사용 방법을 암시해야 한다
- **직접 조작**: 사용자가 객체를 직접 조작하는 느낌을 줄수록 만족감 증가
- **오류 예방 > 오류 복구**: 잘못된 행동을 하기 어렵게 설계하는 것이 우선
- **일관성**: 같은 액션은 언제나 같은 결과를 낸다

## 판단 기준

| 인터랙션 유형 | 피드백 방식 | 응답 시간 기준 |
|-------------|-----------|-------------|
| 버튼 클릭 | 색상 변화 + 눌림 효과 | 즉각 (~100ms) |
| 데이터 로딩 | 스켈레톤 or 스피너 | 300ms 이상 시 표시 |
| 폼 제출 | 로딩 → 성공/에러 | 진행 상태 표시 |
| 삭제/위험 액션 | 확인 다이얼로그 | 되돌릴 수 없음 명시 |
| 드래그&드롭 | 드롭 영역 하이라이트 | 실시간 반응 |

## 실전 예시

**상태 전이 설계 (버튼 제출)**
```
Default → Hover → Active(clicked) → Loading → Success / Error
         ↑색상 변화  ↑눌림 효과    ↑스피너     ↑결과 피드백
```

**인터랙션 명세 템플릿**
- 트리거: 어떤 사용자 행동인가?
- 조건: 어떤 상태일 때 발동하는가?
- 피드백: 즉각적으로 무엇이 변하는가?
- 결과: 최종적으로 어떤 상태가 되는가?
- 실패 시: 오류를 어떻게 처리하는가?

## 안티패턴

- 클릭 후 아무 반응 없음 → 사용자가 다시 클릭 반복
- 비파괴적 액션에 확인 다이얼로그 남발 → 클릭 피로
- Hover state 없이 클릭 가능 요소 설계
- 로딩 중 버튼 비활성화 미처리 → 중복 제출
- 성공/실패 메시지 자동 사라짐 너무 빠름 (2초 미만)

## 실전 팁

- 모든 인터랙션을 Flow 다이어그램으로 먼저 정의
- Edge case (느린 네트워크, 빈 결과, 최대 입력 등) 시나리오 필수 포함
- 손가락 탭 target 크기: 최소 44x44px (Apple HIG 기준)
- 실제 기기에서 직접 테스트. 프로토타입 툴에서는 느끼지 못하는 것들이 있음
- 인터랙션 설계 전 "사용자가 이 순간 무엇을 기대하는가?" 자문

## 심화: 인터랙션 패턴 라이브러리

### 자주 쓰이는 인터랙션 패턴
**Optimistic UI**
- 서버 응답 전에 성공 상태를 먼저 표시
- 실패 시 롤백 + 오류 메시지
- 적용: 좋아요, 팔로우, 간단한 설정 변경

**Progressive Disclosure**
- 처음엔 핵심 정보만, 사용자 요청 시 추가 정보 표시
- 적용: 설정 화면의 "고급 설정", 상품 설명의 "더 보기"

**Skeleton Screen**
- 로딩 중 실제 레이아웃과 유사한 회색 블록 표시
- 빈 화면보다 체감 속도가 빠르게 느껴짐
- 적용: 피드, 카드 리스트, 대시보드

**Infinite Scroll vs Pagination**
- Infinite Scroll: 탐색적 콘텐츠 소비 (SNS 피드)
- Pagination: 목표 지향 탐색, 특정 페이지로 돌아가야 할 때

### 인터랙션 명세 템플릿
```
액션: [무엇을 클릭/터치/입력하는가]
조건: [어떤 상태에서 작동하는가]
즉각 피드백: [0~100ms 내 시각적 변화]
처리 중 상태: [로딩 표시 방식]
성공 결과: [완료 후 상태]
실패 처리: [오류 메시지 + 복구 방법]
접근성: [키보드 동작, aria 속성]
```

### 응답 시간 심리학
- 0~100ms: 즉각적으로 느껴짐 (hover, 클릭 피드백)
- 100~300ms: 약간의 지연감, 피드백 필수
- 300ms~1s: 로딩 인디케이터 표시 시작
- 1s~5s: 진행 상태 표시 (%)
- 5s 이상: 백그라운드 처리 + 완료 알림 패턴 고려
