# 디자인 시스템 — 확장 가능한 일관성의 기반

> 디자인 시스템은 컴포넌트 라이브러리가 아니다. 의사결정의 언어를 통일하는 것이다.

## 핵심 원칙

- **Single Source of Truth**: 디자인 결정은 한 곳에서 관리되고 모든 산출물에 반영된다
- **추상화 계층**: 토큰 → 컴포넌트 → 패턴 → 템플릿 순으로 구체화된다
- **문서화는 시스템의 일부**: 컴포넌트만큼 사용 지침이 중요하다
- **진화 가능성**: 변경에 유연하고, 하위 호환성을 고려한 버저닝이 필요하다
- **실용성 우선**: 완벽한 시스템보다 팀이 실제로 쓰는 시스템이 낫다

## 판단 기준

- 새 화면을 만들 때 기존 컴포넌트 조합으로 80% 이상 해결 가능한가?
- 디자이너와 개발자가 같은 이름으로 같은 것을 부르는가?
- 컴포넌트 변경 시 영향 범위를 사전에 파악할 수 있는가?
- 온보딩된 신규 팀원이 1주일 안에 시스템을 활용할 수 있는가?

## 실전 예시

**계층 구조**
```
Foundation (토큰)
  └── color, spacing, typography, elevation, motion

Components (원자/분자)
  └── Button, Input, Badge, Card, Modal

Patterns (패턴)
  └── Form layout, Empty state, Error handling

Templates
  └── Dashboard, Settings page, Onboarding flow
```

**컴포넌트 문서 필수 항목**
- 사용 목적 (When to use / When NOT to use)
- Props/Variants 명세
- 접근성 고려사항
- 실제 사용 예시 코드

## 안티패턴

- 컴포넌트를 만들되 가이드라인 없이 배포 → 제멋대로 사용됨
- 유사한 컴포넌트 중복 생성 (Button, PrimaryButton, ActionButton...)
- 디자이너만 쓰는 시스템, 개발자와 싱크 없음
- 시스템 업데이트를 멈추고 레거시 방치
- 모든 케이스를 커버하려다 과도하게 복잡해진 컴포넌트

## 실전 팁

- 시스템 구축 전 기존 UI를 인벤토리로 만들어 중복 파악부터 시작
- 첫 버전은 3개 컴포넌트로 시작해도 충분. 점진적으로 확장
- Changelog를 반드시 유지. 변경 이유까지 기록
- 디자인 토큰에 시맨틱 이름 사용: `color-primary` (O) / `blue-500` (X)
- 격월 시스템 리뷰 세션으로 팀 피드백 수렴

## 심화: 디자인 시스템 구축 로드맵

### 단계별 구축 전략
**Phase 1: 기반 (1~2개월)**
- 컬러, 타이포그래피, 간격 토큰 정의
- 핵심 컴포넌트 5~10개 (Button, Input, Card, Modal, Badge)
- Figma 라이브러리 초기 구성
- 기본 문서화 (Notion or Storybook)

**Phase 2: 확장 (2~4개월)**
- 컴포넌트 30~50개로 확장
- 패턴 라이브러리 추가 (Empty state, Error page, Form layout)
- 개발 코드와 싱크 (Storybook + Figma)
- 버전 관리 시작 (Semantic Versioning)

**Phase 3: 성숙 (지속)**
- 기여 프로세스 (팀원이 컴포넌트 제안/기여 가능)
- 접근성 감사 및 개선
- 성능 모니터링 (번들 크기, 렌더 속도)
- 디자인 시스템 로드맵 공개

### 거버넌스 모델
**연방형 (Federated)**
- 중앙 코어 팀 + 제품팀 기여자
- 대규모 조직에 적합
- 코어: 기반 컴포넌트, 가이드라인
- 제품팀: 도메인별 컴포넌트 기여

**중앙 집중형 (Centralized)**
- 전담 디자인 시스템 팀
- 일관성 높음, 병목 가능성 있음

### 성공 지표
- 시스템 채택률: 제품 UI의 몇 %가 시스템 사용
- 컴포넌트 재사용률
- 디자인-개발 핸드오프 시간 단축
- 접근성 오류 수 감소
- 신규 기능 개발 속도 향상

### 디자인 시스템 문서화 도구
- **Storybook**: 컴포넌트 개발/문서화 표준
- **Zeroheight**: 디자인 가이드라인 퍼블리싱
- **Supernova**: Figma → 코드 토큰 자동화
- **Notion**: 의사결정 로그, 회의록
- **GitHub**: 코드 컴포넌트 버전 관리
