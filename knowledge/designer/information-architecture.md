# 정보 구조 (IA) — 사용자가 길을 잃지 않는 구조 설계

> IA가 좋으면 사용자는 네비게이션을 의식하지 않는다. 나쁘면 검색부터 시작한다.

## 핵심 원칙

- **사용자 멘탈 모델**: 사용자가 이미 알고 있는 분류 체계를 따른다
- **발견 가능성**: 기능이 어디 있는지 예측 가능해야 한다
- **최소 클릭**: 핵심 기능까지 3클릭 이내 도달 (경험 법칙)
- **스케일 고려**: 기능이 추가될 때 구조가 무너지지 않아야 한다
- **레이블 명확성**: 메뉴명은 내부 용어가 아닌 사용자 언어로

## 판단 기준

**IA 구조 유형**
| 유형 | 특징 | 적합한 경우 |
|------|-----|-----------|
| 계층형 | 트리 구조, 깊이 존재 | 콘텐츠가 명확히 분류되는 경우 |
| 순차형 | 단계적 진행 | 온보딩, 구매 프로세스 |
| 매트릭스형 | 여러 축으로 탐색 | 전자상거래, 필터 기반 탐색 |
| 네트워크형 | 자유 연결 | 위키, 지식 베이스 |

## 실전 예시

**카드 소팅으로 IA 검증**
1. 콘텐츠/기능 목록을 카드로 제작
2. 참여자 5~10명에게 그룹핑 요청
3. 공통 패턴 분석 → 메뉴 구조 초안 도출
4. Tree Test로 실제 탐색 경로 검증

**레이블 개선 예시**
- X "리소스 관리" → O "파일 및 문서"
- X "컨피규레이션" → O "설정"
- X "마이 페이지" → O "내 계정"
- X "솔루션" → O "기능 / 가격 / 고객 사례"

## 안티패턴

- 조직 구조를 그대로 메뉴 구조로 사용 (내부 사일로 노출)
- 동일한 기능이 여러 메뉴에 중복 배치
- 메뉴 뎁스 4단계 이상 (사용자가 길을 잃음)
- 메뉴에 모든 기능 나열 → 탐색 과부하
- IA 결정을 리서치 없이 직관으로만 처리

## 실전 팁

- OptimalSort(카드 소팅), Treejack(트리 테스트) 도구 활용
- 사이트맵은 항상 최신 상태 유지 (Figma FigJam에서 팀 공유)
- 새 기능 추가 전 "기존 구조의 어디에 들어가는가?" 먼저 정의
- 검색 기능은 IA 실패를 보완하는 수단이지 대체제가 아님
- 사용자 인터뷰에서 "어디서 찾을 것 같으세요?" 질문을 자주 활용

## 심화: IA 설계 프로세스

### 단계별 IA 구축
1. **인벤토리**: 현재 존재하는 모든 콘텐츠/기능 목록화
2. **사용자 리서치**: 멘탈 모델 인터뷰, 카드 소팅
3. **구조 설계**: 그룹핑, 계층 정의, 레이블 작성
4. **검증**: Tree Test로 탐색 성공률 측정
5. **반영**: 결과 기반 구조 수정

### 사이트맵 작성 원칙
- 모든 페이지/화면 포함 (숨겨진 페이지 포함)
- 접근 경로 명시 (어디서 진입 가능한가)
- 권한 레벨 표시 (로그인 필요 여부 등)
- 마지막 업데이트 날짜 기록

### IA 검증 지표
- Task Success Rate: 사용자가 목표를 찾는 성공률 (목표: 80% 이상)
- Time on Task: 특정 콘텐츠에 도달하는 시간
- Directness: 직접 탐색 vs 우회 탐색 비율
- Lostness Score: 방황 정도를 수치화

### 레이블 테스트 방법
- **첫 번째 클릭 테스트**: 사용자가 처음 어디를 클릭하는지
- **5초 테스트**: 홈페이지를 5초 보여주고 어떤 서비스인지 물어보기
- **트리 테스트**: 실제 탐색 경로 추적

### 도구 추천
- Optimal Workshop: 카드 소팅 + 트리 테스트 통합 플랫폼
- Maze: 프로토타입 기반 네비게이션 테스트
- FigJam: 팀 협업 사이트맵 작성
