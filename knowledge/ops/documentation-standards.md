# Documentation Standards — IronAct Ops Knowledge

## 개요
모든 운영 문서의 일관성, 접근성, 유지보수성을 보장하는 표준 정의.

## 문서화 철학

> "기억에 의존하지 않는다. 모든 중요한 것은 기록된다."

좋은 문서는:
- 작성자 없이도 이해 가능
- 업데이트하기 쉬움
- 필요할 때 빠르게 찾을 수 있음

## 문서 유형 및 구조

### 운영 문서 분류
| 유형 | 설명 | 예시 |
|------|------|------|
| 프로세스 문서 | 반복 작업 수행 방법 | SOP, 체크리스트 |
| 참조 문서 | 정보 저장소 | 가이드라인, 정책 |
| 프로젝트 문서 | 특정 프로젝트 기록 | 헌장, 회의록, 보고서 |
| 클라이언트 문서 | 클라이언트별 정보 | 브리프, SLA, 연락처 |

### Notion 폴더 구조
```
운영 (Ops)
├── 프로세스
│   ├── 표준 운영 절차 (SOP)
│   └── 체크리스트 라이브러리
├── 클라이언트
│   └── [클라이언트명]
│       ├── 계약/SLA
│       ├── 브리프
│       └── 납품 이력
├── 프로젝트
│   └── [프로젝트명]
├── 보고서
│   ├── 주간
│   ├── 월간
│   └── QBR
├── 리소스
│   ├── 벤더 목록
│   └── 도구 가이드
└── 아카이브
```

## 문서 작성 표준

### 파일 네이밍
```
[날짜]-[카테고리]-[설명]-[버전]
예: 2026-03-20-client-onboarding-checklist-v1.2
```

### 문서 헤더 (모든 공식 문서)
```
제목: [문서명]
버전: vX.X
작성일: YYYY-MM-DD
최종 수정: YYYY-MM-DD
작성자: [이름]
검토자: [이름]
상태: 초안 / 검토 중 / 승인 / 폐기
```

### 마크다운 작성 규칙
- H1: 문서 제목 (1개만)
- H2: 주요 섹션
- H3: 소섹션
- 테이블: 비교/목록 정보
- 코드 블록: 명령어, 템플릿
- 강조: **굵게** (중요), _기울임_ (용어)

## 버전 관리

### 버전 넘버링
```
Major.Minor
1.0 → 1.1 (마이너 수정)
1.1 → 2.0 (구조적 변경)
```

### 변경 이력 기록
```
## 변경 이력
| 버전 | 날짜 | 변경 내용 | 작성자 |
|------|------|-----------|--------|
| 1.0 | 2026-01-01 | 최초 작성 | - |
| 1.1 | 2026-03-01 | 섹션 X 업데이트 | - |
```

## 문서 유지보수

### 정기 리뷰 주기
| 문서 유형 | 리뷰 주기 |
|-----------|-----------|
| 프로세스 문서 | 분기별 |
| 클라이언트 정보 | 변경 시마다 |
| 보고서 | 아카이브 (수정 없음) |
| 벤더 정보 | 반기별 |

### 문서 폐기 기준
- 사용 안 된 지 1년 이상
- 대체 문서 존재
- 정보가 더 이상 유효하지 않음

→ 삭제 대신 `아카이브` 폴더로 이동

## 문서 접근성

### 검색 가능성 높이기
- Notion 태그 활용 (클라이언트명, 카테고리, 상태)
- 제목에 핵심 키워드 포함
- 관련 문서 상호 링크

### 공유 수준 설정
| 수준 | 접근 범위 |
|------|-----------|
| 전체 공개 | 전 팀원 |
| 팀 한정 | 특정 팀만 |
| 비공개 | 작성자 + 지정인 |
| 클라이언트 공유 | 클라이언트 + 내부 |

- [ ] 폐기 대상 문서 아카이빙
- [ ] Notion 구조 신규 팀원에게 안내
