# Escalation — IronAct Ops Knowledge

## 개요
이슈를 적절한 타이밍에 적절한 수준으로 상위 의사결정자에게 전달하는 체계.

## 에스컬레이션이란?
현재 레벨에서 해결 불가능한 이슈를 상위 권한 또는 전문성을 가진 주체로 이전하는 것.

## 에스컬레이션 트리거 (언제 올려야 하는가)

### 즉시 에스컬레이션
- SLA 위반 확정 (복구 불가)
- 클라이언트 계약 해지 언급
- 주요 클라이언트 경영진 직접 컨택
- 법적/법률적 이슈 발생
- 팀원 심각한 갈등 또는 안전 이슈
- 데이터 유실 / 보안 침해

### 24시간 내 에스컬레이션
- 블로커가 24시간 이상 해소 안 됨
- 클라이언트 강한 불만 (공식 항의)
- 프로젝트 일정이 30% 이상 지연
- 예산 초과 임박

### 주간 에스컬레이션 (정기 보고)
- 지속적인 KPI 미달
- 반복적인 프로세스 문제
- 리소스 부족 구조적 문제

## 에스컬레이션 경로

```
팀원
  ↓ (즉시)
Ops Lead / 팀장
  ↓ (Level 2 이상)
Director / VP
  ↓ (Level 1 또는 경영 사안)
CEO / 경영진
```

### 클라이언트 불만 에스컬레이션
```
담당 AM
  ↓ (불만 강도: 높음)
Ops Lead
  ↓ (계약 해지 위기)
팀장 + 경영진
```

## 에스컬레이션 방법

### 에스컬레이션 메시지 구조 (SBAR)
```
S - Situation (상황):
  무슨 일이 발생했는가? 언제?

B - Background (배경):
  맥락은? 어디까지 진행됐는가?

A - Assessment (평가):
  현재 영향과 위험도는?

R - Recommendation (권고):
  내가 제안하는 다음 액션은?
```

### 예시
```
[긴급 에스컬레이션] 클라이언트 A사 납품 지연

S: A사 주간 콘텐츠 납품이 오늘 오전까지였으나 제작팀 이슈로 오늘 내 납품 불가 확인됨 (3시간 전)

B: 제작팀 담당자 급병으로 인한 공백. 대체 인력 없는 상황. 이미 클라이언트에 오전 중 납품 약속.

A: SLA 위반 확정. 클라이언트가 타이트한 발행 일정을 갖고 있어 비즈니스 임팩트 가능성 높음. 클라이언트 건강도 Yellow 상태.

R: (1) 즉시 클라이언트에 상황 사전 통보 (2) 오늘 저녁까지 대체 납품 방안 모색 (3) Ops Lead 또는 팀장 직접 클라이언트 연락 고려. 결정 권한 요청.
```

## 에스컬레이션 후 처리

### 에스컬레이션 받은 사람의 책임
- 30분 내 접수 확인
- 액션 또는 추가 정보 요청
- 결과를 에스컬레이션한 사람에게 피드백

### 에스컬레이션 결과 기록
- Linear 이슈에 에스컬레이션 태그
- 결과 및 조치 사항 기록
- 재발 방지 여부 확인

## 에스컬레이션 문화

### 건강한 에스컬레이션
- 에스컬레이션은 실패가 아니라 시스템이 작동하는 것
- 너무 늦게 올리는 것이 너무 일찍 올리는 것보다 위험
- "내가 해결해야 한다"는 자존심이 에스컬레이션을 막아선 안 됨

### 안티패턴
- ❌ 혼자 해결하려다 시간 낭비
- ❌ 에스컬레이션 전 충분한 정보 수집 없이 보고
- ❌ 솔루션 없이 문제만 보고
- ❌ 에스컬레이션 결과를 팀에 공유 안 함
