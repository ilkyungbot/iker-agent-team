# Scaling — IronAct Ops Knowledge

## 개요
비즈니스 성장에 맞춰 운영 역량을 확장하는 전략과 실행 방법.

## 스케일링의 원칙

> "스케일링은 더 많은 사람을 추가하는 것이 아니라, 시스템이 사람을 대신할 수 있게 만드는 것이다."

## 스케일링 트리거 (언제 확장이 필요한가)

### 용량 기반 트리거
- 팀 용량이 2개월 연속 85%+ 유지
- 납품 지연이 구조적으로 반복
- 클라이언트 만족도 하락 (리소스 부족 원인)

### 품질 기반 트리거
- 품질 문제가 증가세
- 한 사람에게 지식이 집중
- QC 단계가 병목

### 성장 기반 트리거
- 신규 클라이언트 획득 속도가 팀 용량 초과
- 새로운 서비스 라인 추가
- 지역/언어 확장

## 스케일링 레버 (4가지 방법)

### 1. 프로세스 표준화
가장 먼저, 가장 중요한 레버.
```
표준화 전: 사람마다 다른 방식 → 품질 편차
표준화 후: 동일한 프로세스 → 일관된 품질
```
- 모든 반복 작업 SOP 작성
- 체크리스트 의무화
- 템플릿 라이브러리 구축

### 2. 자동화
참조: automation.md
- 반복 작업 자동화
- 자동 알림/보고
- 데이터 수집 자동화

### 3. 팀 확장
신중하게, 준비된 후에.
```
채용 전 확인:
□ 역할 명확히 정의됐는가?
□ 온보딩 프로세스 준비됐는가?
□ 역할 없이 3개월 이상 버텼는가?
□ 채용 예산 확보됐는가?
```

#### 채용 순서
1. 핵심 역량 공백 먼저
2. 가장 큰 병목 지점
3. 성장을 막는 제약

### 4. 외주/파트너 활용
- 일시적 수요 급증 대응
- 전문 스킬 단기 확보
- 테스트 후 내재화 여부 결정
참조: vendor-management.md

## 스케일링 로드맵

### Phase 1: 기반 구축 (현재~다음 분기)
- 핵심 프로세스 100% 문서화
- 반복 업무 자동화 50% 달성
- 온보딩 매뉴얼 완성

### Phase 2: 확장 준비 (다음 분기~6개월)
- 팀 용량 여유 20% 확보
- 핵심 인력 지식 이전 완료
- 첫 외주 파트너 검증

### Phase 3: 성장 실행 (6개월~1년)
- 팀 확장 또는 외주 체계 구축
- 신규 서비스 라인 운영 역량 확보
- 품질 KPI 동일 수준 유지 확인

## 스케일링 리스크 관리

### 품질 저하 리스크
- QC 프로세스 스케일링과 동시에
- 품질 KPI 더 촘촘히 모니터링
- 확장 초기 3개월 집중 점검

### 문화 희석 리스크
- 팀이 커질수록 의도적 문화 구축 필요
- 신규 팀원 온보딩에 문화 임베딩
- 핵심 가치 반복 강조

### 속도 저하 리스크
- 프로세스 과다 → 의사결정 느려짐
- 불필요한 승인 단계 제거
- 위임 범위 확대

## 스케일링 성공 지표
```
이전과 동일한 품질 KPI + 더 높은 처리량
= 성공적인 스케일링
```
