# Communication — IronAct Ops Knowledge

## 개요
내부 팀 및 클라이언트와의 효과적인 커뮤니케이션 기준과 프로토콜.

## 채널별 용도

### Slack
| 채널 유형 | 용도 | 응답 기대 시간 |
|-----------|------|----------------|
| #general | 전사 공지 | 당일 |
| #project-[이름] | 프로젝트별 | 4시간 내 |
| #ops | 운영팀 내부 | 2시간 내 |
| DM | 긴급/1:1 | 1시간 내 |

### Slack 메시지 작성 원칙
1. 메시지 앞에 맥락 제공 (한 줄 요약)
2. 행동 요청은 `@멘션` + 명확한 요청사항
3. 긴급도 표기: `[긴급]`, `[FYI]`, `[액션필요]`
4. 스레드 활용 (메인 채널 오염 방지)

### 이메일
- 공식 외부 커뮤니케이션
- 계약/법적 사항
- 미팅 초대 및 확인
- 응답 기대 시간: 업무일 24시간 내

### Notion
- 영구 문서화
- 프로젝트 산출물
- 회의록
- 프로세스 문서

### Google Calendar
- 모든 미팅은 캘린더 이벤트 생성
- 최소 24시간 전 초대
- 안건(Agenda) 사전 공유

## 미팅 커뮤니케이션 원칙

### 미팅 전
- 목적(Purpose) 명확화
- 안건 사전 공유 (최소 24시간 전)
- 필요 자료 준비
- 참석자 적정성 검토 (2-pizza rule)

### 미팅 중
- 시작 시 목표 재확인
- 서기(Notetaker) 지정
- 결정 사항 즉시 기록
- 시간 관리 철저

### 미팅 후
- 24시간 내 회의록 공유
- 액션 아이템: 담당자 + 마감일 명시
- Notion에 기록 저장

## 보고 커뮤니케이션

### 상향 보고 (경영진)
```
형식: 결론 먼저 → 근거 → 다음 액션
길이: 1페이지 이내
주기: 주간 (금요일 EOD)
```

### 수평 보고 (팀 간)
```
형식: 현황 → 이슈 → 지원 필요사항
길이: 자유 (핵심만)
주기: 필요 시
```

### 하향 보고 (팀원에게)
```
형식: 배경 → 목표 → 방법 → 기대 결과
강조: Why (이유)를 반드시 포함
```

## 어려운 대화 처리

### 나쁜 소식 전달 (Bad News)
1. 즉시 전달 (숨기지 않는다)
2. 사실(Fact)만 먼저
3. 영향(Impact) 설명
4. 이미 취한 조치 공유
5. 다음 단계 제시

### 의견 충돌 해소
1. 상대방 입장 먼저 경청
2. 공통 목표 재확인
3. 데이터 기반으로 논의
4. 합의 안 되면 에스컬레이션

## 커뮤니케이션 안티패턴
- ❌ 중요 사항을 DM으로만 처리 (공식 채널 병행 필수)
- ❌ 답장 없이 read만 (반응 필수: 확인 이모지라도)
- ❌ 블로커를 혼자 끌어안기 (즉시 공유)
- ❌ 미팅 없이 중요 결정 (문서화 필수)
- ❌ 두루뭉술한 마감일 (항상 날짜 명시)

## 외부 클라이언트 커뮤니케이션

- 항상 다음 스텝 명시
- 긍정적 표현 우선

- 긴급 문의: 4시간
