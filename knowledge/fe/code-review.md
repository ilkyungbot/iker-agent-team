# Code Review — 코드 리뷰 가이드

> 코드 리뷰는 버그를 찾는 게 아니라 더 나은 코드를 함께 만드는 과정이다.

## 핵심 원칙
- **사람이 아닌 코드 리뷰**: 코드를 비판하되 사람을 비판하지 말 것
- **의도 파악 우선**: 왜 이렇게 짰는지 먼저 이해
- **건설적 제안**: 문제 지적 + 개선 방향 함께 제시
- **PR 크기 제한**: 300-400줄 초과 시 분리 요청

## 리뷰 체크리스트

### 작성자 (PR 올리기 전)
- [ ] PR 설명에 변경 이유와 영향 범위 작성
- [ ] 스크린샷/GIF 첨부 (UI 변경 시)
- [ ] 셀프 리뷰 완료
- [ ] 관련 테스트 작성
- [ ] 300줄 이내 (불가 시 이유 설명)

### 리뷰어 체크리스트
- [ ] 버그 가능성 (엣지 케이스, 타입 안전성)
- [ ] 성능 (불필요한 리렌더, N+1, 무한 루프)
- [ ] 보안 (XSS, 민감정보 노출)
- [ ] 접근성 (ARIA, 키보드 네비게이션)
- [ ] 테스트 커버리지 (중요 경로 커버)

## 코드 예시

✅ 명확한 PR 설명 템플릿
```markdown
## 변경 사항
- 사용자 프로필 수정 폼 추가
- 이메일 중복 검사 API 연동

## 변경 이유
- PLT-1234: 사용자가 프로필을 수정할 수 없는 문제

## 테스트 방법
1. /settings/profile 접속
2. 이름/이메일 수정 후 저장
3. 이미 사용 중인 이메일 입력 시 에러 확인

## 스크린샷
(이미지 첨부)

## 영향 범위
- `/settings/profile` 페이지만 영향
```

✅ 건설적인 리뷰 댓글 예시
```
// ❌ 비건설적
"이 코드 왜 이렇게 짰어요?"

// ✅ 건설적
"이 부분에서 items가 빈 배열일 때 undefined를 반환할 수 있을 것 같아요.
`items[0]?.name ?? ''` 로 처리하는 건 어떨까요?"

// ✅ 칭찬도 적극적으로
"이 패턴 처음 봤는데 훨씬 깔끔하네요. 저도 앞으로 써야겠어요!"
```

✅ 리뷰 댓글 분류 (Conventional Comments)
```
// 필수 수정
blocking: 이 방식은 XSS 취약점이 있어요. dangerouslySetInnerHTML 사용 전 DOMPurify sanitize 필요합니다.

// 제안 (선택)
suggestion: 가독성을 위해 early return 패턴을 써볼 수도 있을 것 같아요.

// 질문
question: 이 경우 왜 useCallback이 필요한가요? 의존성 배열이 비어있으면 최초 1회만 생성되는 거 맞나요?

// 칭찬
praise: 에러 처리를 이렇게 세분화하니 디버깅이 훨씬 쉬울 것 같아요!
```

## 실전 팁
- 24시간 내 첫 리뷰 응답 목표
- 댓글 30개 넘으면 대화로 해결하는 게 빠름
- `nit:` 접두사로 사소한 지적 구분 (반영 안 해도 됨)

## 주의사항
- 스타일 논쟁 (탭 vs 스페이스)은 linter에 위임
- "제가 이렇게 썼을 것 같아요"는 좋은 피드백이 아님 — 이유 설명 필수
- 리뷰 지연은 팀 속도 저하 — 하루 1-2회 리뷰 확인 리듬 만들기
