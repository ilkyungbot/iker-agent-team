# PRD 작성법 — PRD 작성법과 템플릿
> 핵심 메시지: "PRD는 무엇을 만드는지가 아니라 왜 만드는지를 설명하는 문서다"

## 핵심 원칙
- PRD는 팀 정렬 도구, 지시 도구가 아니다
- "왜"가 "무엇"보다 중요: 목적을 이해한 팀이 더 좋은 솔루션을 찾는다
- 린 PRD: 필요한 정보만 담되, 팀이 자율적으로 결정할 공간을 남긴다
- 살아있는 문서: 작성 후 지속적으로 업데이트

## 판단 기준
- 이 PRD를 읽은 팀원이 "왜 이것을 만드는가"를 설명할 수 있는가?
- 성공 기준이 측정 가능한 형태로 정의되어 있는가?
- 범위 밖(Out of Scope)이 명확히 정의되어 있는가?
- 구현 세부 사항을 과도하게 지시하지 않는가?

## 실전 예시 (PRD 템플릿)

### 표준 PRD 구조
```
# [기능명] PRD

## 요약 (TL;DR)
- 한 문단으로 이 기능의 목적과 기대 효과 설명

## 배경 및 맥락
- 현재 상황과 문제점
- 왜 지금 이것을 해야 하는가
- 관련 데이터 및 리서치 인사이트

## 목표 (Goals)
- 비즈니스 목표: [측정 가능한 지표]
- 사용자 목표: [사용자가 달성하고 싶은 것]
- 비목표 (Non-goals): [이번에 하지 않을 것]

## 사용자 스토리
- As a [사용자 유형], I want to [행동], so that [이유]

## 성공 지표 (Success Metrics)
- Primary: [핵심 지표 1개]
- Secondary: [보조 지표 2-3개]
- 가드레일 지표: [나빠지면 안 되는 지표]

## 솔루션 개요
- 제안하는 솔루션 (탐색한 대안 포함)
- 선택 이유
- 핵심 사용자 플로우

## 기능 요구사항
- [기능 1]: 상세 설명
- [기능 2]: 상세 설명

## 범위 외 (Out of Scope)
- [명시적으로 제외하는 항목들]

## 오픈 이슈 / 질문
- [결정되지 않은 사항들]

## 참고 자료
- 사용자 인터뷰 링크
- 경쟁사 분석
- 디자인 목업
```

### 사용자 스토리 작성 기준 (INVEST)
- **I**ndependent: 다른 스토리와 독립적
- **N**egotiable: 구현 방법은 협상 가능
- **V**aluable: 사용자에게 가치 제공
- **E**stimable: 크기 추정 가능
- **S**mall: 한 스프린트 내 완료 가능
- **T**estable: 검증 가능한 완료 기준

### 수락 기준 (Acceptance Criteria) 작성법
Given [사전 조건] When [행동] Then [기대 결과]

예시: Given 로그인한 사용자가 When 결제 버튼을 클릭하면 Then 결제 모달이 표시된다

## 안티패턴
- 100페이지 PRD: 읽지 않는 문서는 의미 없다
- UI 스펙을 PRD에 포함: 디자이너의 영역 침범
- "어떻게" 구현할지 지시: 엔지니어의 창의성 차단
- 이해관계자 승인을 기다리는 PRD 병목
- 완성 후 서랍에 넣는 PRD: 지속 업데이트 필요

## 실전 팁
- PRD 리뷰는 30분 이내로 제한
- 엔지니어와 디자이너를 PRD 작성 초기부터 참여
- "만약 이 기능이 없다면 어떻게 되는가?" 테스트
- PRD에 경쟁사 벤치마크 스크린샷 포함
- 출시 후 성공 지표 달성 여부를 PRD에 다시 기록 (학습 루프)
