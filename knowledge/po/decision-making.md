# 의사결정 — MECE, 2x2 의사결정 프레임워크
> 핵심 메시지: "좋은 의사결정 프로세스가 좋은 결과보다 중요하다"

## 핵심 원칙
- 의사결정의 질은 결과가 아닌 프로세스로 판단한다
- 충분한 정보가 갖춰지기 전에도 결정해야 할 때가 있다
- 가역적 결정은 빠르게, 비가역적 결정은 신중하게
- 결정 이유를 기록하면 나중에 배울 수 있다

## 판단 기준
- 이 결정을 되돌리는 것이 얼마나 어려운가? (가역성)
- 결정에 필요한 정보의 70%를 갖고 있는가?
- 이해관계자들이 결정 방식에 동의하는가?
- 결정 후 실행할 주체가 명확한가?

## 실전 예시 (프레임워크)

### MECE (Mutually Exclusive, Collectively Exhaustive)
- 상호 배타적: 각 옵션이 겹치지 않음
- 전체 포괄적: 모든 가능한 옵션을 포함
- 활용: 문제 구조화, 옵션 나열, 이슈 트리 작성

이슈 트리 예시:
```
매출 감소
├── 신규 고객 감소
│   ├── 유입 감소 (마케팅 문제)
│   └── 전환율 감소 (제품/UX 문제)
└── 기존 고객 이탈
    ├── 만족도 감소 (제품 문제)
    └── 경쟁사 이동 (가격/기능 문제)
```

### 2x2 의사결정 매트릭스
기준 2가지를 축으로 옵션 배치:
- Impact vs Effort
- Risk vs Value
- Urgency vs Importance (아이젠하워 매트릭스)

### Amazon의 Type 1 / Type 2 결정
- Type 1 (비가역적): 신중하게, 충분한 분석 후 결정
  - 예: 핵심 가격 정책 변경, 주요 파트너십
- Type 2 (가역적): 빠르게, 70% 정보로 결정
  - 예: UI 변경, 새 기능 실험

### DACI 의사결정 모델
- **D**river: 결정을 추진하는 책임자 (1명)
- **A**pprover: 최종 승인자 (1명)
- **C**ontributors: 의견을 제공하는 기여자 (여러 명)
- **I**nformed: 결정 후 통보 받을 사람들

### 6-Page Memo (Amazon 방식)
1. 현황: 해결해야 할 문제/기회
2. 목표: 원하는 결과
3. 고려한 옵션들
4. 권장 사항과 이유
5. 리스크와 완화 방안
6. 성공 측정 방법

### 결정 일지 (Decision Log) 템플릿
```
결정 날짜:
결정 내용:
결정 이유:
고려한 대안:
예상 결과:
결정 주체:
리뷰 예정일:
실제 결과 (6개월 후 기록):
```

## 안티패턴
- Analysis Paralysis: 완벽한 정보를 기다리며 결정 지연
- HiPPO(최고연봉자 의견)에 의한 결정
- 합의에 의한 결정: 모두를 기쁘게 하려다 최선의 결정을 못 함
- 결정 이유 기록 안 함: 6개월 후 왜 이렇게 됐는지 모름
- 결정 책임자 불명확: 모두의 결정은 아무도의 결정

## 실전 팁
- "만약 6개월 후의 나라면 이 결정을 어떻게 평가할까?" 자문
- 최악의 시나리오 테스트: 이 결정이 틀렸을 때의 결과는 감당 가능한가?
- Pre-mortem: "1년 후 이 결정이 실패했다면, 이유는 무엇일까?"
- 결정 속도를 높이려면 의사결정 프레임워크를 팀이 공유
- 중요 결정은 72시간의 숙고 시간 부여
