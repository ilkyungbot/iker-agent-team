# 제품 발견 — 가설→실험→학습 루프
> 핵심 메시지: "빌드하기 전에 발견하라. 발견 비용이 개발 비용보다 100배 싸다"

## 핵심 원칙
- Continuous Discovery: 발견은 분기별 이벤트가 아닌 주간 루틴
- Opportunity Solution Tree: 기회(문제) → 솔루션 → 실험 순서로 접근
- 가정 검증 우선: 가장 위험한 가정을 먼저 테스트
- 빠른 실패: 작게 실패하고 빨리 배우는 것이 목표

## 판단 기준
- 이 실험이 가장 위험한 가정을 검증하는가?
- 2주 안에 결과를 볼 수 있는가?
- 학습이 다음 의사결정에 직접 영향을 주는가?
- 실험 설계 전에 성공 기준을 정했는가?

## 실전 예시 (프레임워크)

### Opportunity Solution Tree (OST) 구성
```
Desired Outcome (목표 성과)
├── Opportunity 1 (고객 문제/니즈)
│   ├── Solution A
│   │   └── Experiment 1
│   └── Solution B
│       └── Experiment 2
└── Opportunity 2
    └── Solution C
        └── Experiment 3
```

### 가정 매핑 (Assumption Mapping)
| 가정 유형 | 설명 | 우선순위 |
|---------|------|--------|
| 고객 가정 | 이 문제가 실제로 존재하는가 | 최우선 |
| 문제 가정 | 고객이 이것을 문제로 인식하는가 | 높음 |
| 솔루션 가정 | 우리 솔루션이 문제를 해결하는가 | 중간 |
| 비즈니스 가정 | 수익화 가능한가 | 중간 |
| 실행 가정 | 우리가 만들 수 있는가 | 낮음 |

### 최소 테스트 유형 선택
- 고객 인터뷰: 문제 존재 여부 검증 (비용 최소)
- 페이퍼 프로토타입: 솔루션 컨셉 검증
- 랜딩 페이지: 수요 검증
- Wizard of Oz: 자동화 전 수동으로 서비스 제공
- 기능 가짜 버튼: 클릭률로 관심도 측정
- 스모크 테스트: 실제 구현 전 구매 의사 확인

### 주간 Discovery 루틴
- 매주 최소 2명 고객 인터뷰
- 인터뷰 후 24시간 내 인사이트 문서화
- 2주마다 OST 업데이트
- 월 1회 Discovery 리뷰 (팀 전체)

## 안티패턴
- 솔루션 먼저, 문제는 나중: 이미 만든 것을 정당화하는 리서치
- 인터뷰에서 솔루션 피칭: 검증이 아닌 설득을 하는 실수
- 한 번의 큰 실험: 여러 작은 실험이 더 많은 학습을 제공
- 확증 편향: 원하는 답을 찾으려는 인터뷰 설계
- Discovery와 Delivery 분리: 팀 전체가 함께 발견해야 함

## 실전 팁
- "어떻게 해결했나요?" 질문으로 현재 대안 파악
- 5 Whys: 표면 문제 뒤의 근본 원인 탐색
- 고객 여정 지도로 전체 경험에서 기회 발굴
- 인터뷰 노트는 직접 인용구를 최대한 보존
- 사용자 스토리: "~로서, ~하고 싶다, ~하기 위해"
