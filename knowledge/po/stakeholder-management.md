# 이해관계자 관리 — 이해관계자 식별과 관리
> 핵심 메시지: "이해관계자를 관리하지 않으면 이해관계자가 당신을 관리한다"

## 핵심 원칙
- 이해관계자의 요청 뒤에 있는 니즈를 파악한다
- 모든 요청에 "예스"는 모든 요청에 "노"와 같다
- 기대치 관리가 결과물 관리보다 중요할 때가 많다
- 나쁜 소식은 빨리, 직접, 해결책과 함께 전달

## 판단 기준
- 각 이해관계자의 성공 기준이 무엇인지 알고 있는가?
- 이해관계자의 기대치가 현실적으로 설정되어 있는가?
- 정기적으로 업데이트를 제공하고 있는가?
- 갈등하는 요청들 사이에서 전략에 근거한 결정을 내리는가?

## 실전 예시 (프레임워크)

### 이해관계자 지도 (Stakeholder Map)
```
높은 영향력   │ 긴밀한 관리    │ 충족시켜야    │
             │ (CEO, CTO)    │ 함(투자자)   │
─────────────────────────────────────────────
낮은 영향력   │ 정보 제공      │ 모니터링      │
             │ (영업팀)      │ (외부 파트너) │
             └─────────────────────────────
                높은 관심도     낮은 관심도
```

### 이해관계자 프로파일 템플릿
```
이름/역할:
관심사: [이 사람이 가장 신경 쓰는 것]
성공 기준: [이 사람에게 성공이란?]
소통 선호도: [이메일/회의/슬랙]
소통 주기: [주간/월간]
주요 우려사항:
현재 관계 상태: [우호적/중립/비우호적]
```

### 요청 처리 4단계
1. **이해**: "왜 이 기능이 필요한가?" 근본 니즈 파악
2. **검토**: 전략과의 정합성, 우선순위, 현실 가능성 평가
3. **결정 및 이유**: 수락/부분 수락/거절과 명확한 이유
4. **대안 제시**: 거절 시 대안적 해결책 또는 타임라인 제시

### "No"를 말하는 방법
나쁜 예: "백로그가 가득 차서 못 합니다"
좋은 예: "현재 저희 최우선 목표는 X입니다. 이 요청이 X보다 중요하다면 기존 계획을 조정해야 합니다. 그 트레이드오프를 검토하고 싶으시다면 논의할 수 있습니다"

### 이해관계자 업데이트 리듬
| 이해관계자 | 주기 | 형식 | 내용 |
|---------|-----|-----|-----|
| CEO | 월간 | 1-pager | 핵심 지표, 전략 진행 |
| CTO | 주간 | 슬랙 | 기술 결정, 블로커 |
| 영업팀 | 격주 | 이메일 | 로드맵 현황, 고객 약속 가능 항목 |
| 고객 | 분기 | 뉴스레터 | 출시 항목, 다음 방향 |

### 갈등하는 요청 처리
1. 전략 문서로 돌아가기: "우리 전략에 따르면..."
2. 데이터 제시: "현재 지표를 보면..."
3. 트레이드오프 명시화: "A를 하면 B는 못 합니다"
4. 의사결정 권한자에게 에스컬레이션: "이 결정은 제 권한 밖입니다"

## 안티패턴
- 모든 요청을 수락하는 "예스맨" PO
- 이해관계자 요청을 무시하거나 회피
- 나쁜 소식을 늦게 전달
- 이해관계자마다 다른 로드맵 약속
- 이해관계자의 신뢰 없이 제품 결정 강행

## 실전 팁
- 분기 시작 시 이해관계자별 기대치 명시적으로 정렬
- "이 요청의 배경을 더 이해하고 싶습니다" - 항상 니즈 먼저 파악
- 로드맵 변경 시 영향 받는 이해관계자에게 먼저 1:1 알림
- 이해관계자를 Discovery 인터뷰에 참여시켜 공감대 형성
- 감사 표현: 좋은 피드백과 지원에 대한 인정
