# A/B 테스팅 — 실험 설계와 통계적 유의성
> 핵심 메시지: "좋은 실험 하나가 백 번의 토론을 대체한다"

## 핵심 원칙
- 실험 전 가설과 성공 기준을 명시적으로 정한다
- 통계적 유의성 없이 결론 내리지 않는다
- 실험 기간 중 중간 결과를 보고 멈추지 않는다 (Peeking Problem)
- 한 번에 하나의 변수만 변경한다

## 판단 기준
- 이 실험이 의미 있는 결과를 도출할 충분한 샘플 크기를 갖는가?
- 실험 기간이 충분한가? (최소 1-2주, 계절성 고려)
- 성공/실패 기준이 사전에 정의되어 있는가?
- 실험 결과가 어떻든 실행 가능한 결론을 낼 수 있는가?

## 실전 예시 (프레임워크)

### 실험 설계 프로세스
1. 가설 작성: "[변경]을 하면 [타겟 사용자]의 [지표]가 [예상 변화]할 것이다"
2. 기본 지표(Primary Metric) 선정 (1개)
3. 보조 지표(Secondary Metrics) 선정 (2-3개)
4. 가드레일 지표 선정 (나빠지면 안 되는 것)
5. 최소 샘플 크기 계산
6. 실험 기간 결정
7. 실험 실행 및 모니터링
8. 결과 분석 및 결론

### 샘플 크기 계산 기준
- Significance Level (α): 0.05 (5% 오류 허용)
- Statistical Power (1-β): 0.80 (80% 검출력)
- Minimum Detectable Effect (MDE): 감지하고 싶은 최소 효과 크기
- 공식: n ≈ 16σ² / δ² (σ: 표준편차, δ: MDE)
- 실용 도구: Evan's A/B Test Calculator 사용

### 실험 설계 문서 템플릿
```
실험명: [명확한 이름]
가설: [구체적인 가설 문장]
배경: [왜 이 실험을 하는가]

실험 설계:
- Control: 현재 상태
- Treatment: 변경 사항

지표:
- Primary: [핵심 지표]
- Secondary: [보조 지표들]
- Guardrail: [지켜야 할 지표]

기술 요구사항:
- 샘플 크기: [계산된 크기]
- 실험 기간: [시작일 ~ 종료일]
- 트래픽 분배: Control 50% / Treatment 50%

성공 기준:
- Primary 지표가 [X]% 이상 개선되고 통계적으로 유의미(p < 0.05)

예상 결과:
- 긍정: [결과가 좋으면 할 일]
- 부정: [결과가 나쁘면 할 일]
- 무효: [차이 없으면 할 일]
```

### 통계 해석 가이드
- p-value < 0.05: 통계적으로 유의미 (5% 미만 우연일 확률)
- Confidence Interval: 효과 크기의 범위 확인
- Practical Significance vs Statistical Significance: 둘 다 필요
- Multiple Testing Problem: 여러 지표 동시 검증 시 Bonferroni 보정

### 흔한 실험 결과 해석
- A 승리 (p < 0.05, 긍정적 방향): A 전면 도입 검토
- B 승리 (p < 0.05, 부정적 방향): 즉시 중단
- 무효 (p > 0.05): 가설 재검토, 다른 접근 시도
- 혼합 결과: 세그먼트별 분석으로 특정 사용자 그룹 효과 확인

## 안티패턴
- Peeking: 중간에 결과 보고 조기 종료
- HIPPO 실험: "CEO가 좋다고 했으니 확인만 하자"
- 너무 많은 동시 실험: 상호작용 효과로 결과 오염
- 실험 결과 무시: 데이터가 불편해도 학습으로 수용
- 운영 비용 무시: 실험 인프라 유지 비용 과소평가

## 실전 팁
- 실험 레지스트리: 진행 중/완료된 실험을 팀이 공유
- Feature Flag 시스템: 빠른 실험 롤아웃/롤백 가능
- 실험 결과 문서화: 성공/실패 모두 기록하여 지식 축적
- 주간 실험 리뷰: 팀이 실험 현황 공유
- "실험 문화": 실패한 실험도 가치 있음을 팀이 인식
