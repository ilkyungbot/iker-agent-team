# 스타트업 운영 — 스타트업 환경에서의 제품 운영
> 핵심 메시지: "스타트업은 빠른 학습 머신이다. PO는 그 학습 속도를 최대화한다"

## 핵심 원칙
- Speed over Perfection: 완벽한 제품보다 빠른 학습
- 자원 제약이 창의성을 만든다: 제약을 활용하라
- PMF를 찾기 전까지 스케일하지 않는다
- 모든 역할을 혼자 해야 할 때도 있다

## 판단 기준
- 지금 하는 일이 PMF 탐색에 기여하는가?
- 이 결정이 1주일 내로 검증 가능한가?
- 팀의 런웨이(생존 기간)를 고려한 결정인가?
- "나중에" 미루는 것이 진짜 우선순위인가, 회피인가?

## 실전 예시 (프레임워크)

### PMF (Product-Market Fit) 측정
**숀 엘리스 테스트**: "이 제품이 사라진다면 어떻게 되나요?"
- 매우 실망 > 40% = PMF 도달
- 매우 실망 < 40% = 계속 탐색

**리텐션 기반 PMF**: 리텐션 곡선이 수평선에서 안정화

**정성적 신호**:
- 고객이 자발적으로 추천
- 요금 인상에도 이탈하지 않음
- "이 없이는 못 살겠다"는 표현

### 스타트업 단계별 PO 포커스
**Pre-PMF (0→1)**
- 핵심 집중: 고객 인터뷰, 빠른 프로토타입
- 지표: 리텐션, NPS, 활성 사용자 품질
- 위험: 너무 많은 기능 빌드, 잘못된 고객 타겟

**Post-PMF (1→10)**
- 핵심 집중: 핵심 경험 최적화, 온보딩 개선
- 지표: CAC, LTV, MRR 성장률
- 위험: PMF 없이 스케일, 기술 부채 누적

**스케일 (10→100)**
- 핵심 집중: 팀 구조화, 프로세스 확립
- 지표: NPS, 이탈률, CLTV
- 위험: 관료화, 속도 저하

### 린 스타트업 실행 루프
```
아이디어 → 빌드 → 제품 → 측정 → 데이터 → 학습 → (반복)
```
각 루프를 최대한 빠르게 (목표: 2주 이내)

### 최소 기능 제품 (MVP) 설계
1. 핵심 고객 문제 1개 선정
2. 그 문제를 해결하는 가장 단순한 방법
3. 측정 가능한 성공 기준 사전 정의
4. 빌드하고 실제 사용자에게 배포
5. 데이터와 피드백 수집
6. 빌드/피벗/중단 결정

### 스타트업 PO의 주간 루틴
| 요일 | 활동 |
|-----|-----|
| 월 | 주간 지표 리뷰, 스프린트 시작 |
| 화 | 고객 인터뷰 (2건) |
| 수 | 팀 스탠드업, PRD 작성 |
| 목 | 이해관계자 업데이트, 1:1 미팅 |
| 금 | 스프린트 리뷰, 레트로스펙티브, 주간 리뷰 작성 |

### 런웨이 인식 의사결정
- 런웨이 < 12개월: 수익화 가속, 비용 절감 우선
- 런웨이 12-24개월: PMF 최적화와 성장 병행
- 런웨이 > 24개월: 장기 전략 투자 가능

## 안티패턴
- Stealth Mode: 출시 전 너무 오랜 시간 투자
- 기능 추가로 PMF 찾기: 핵심 문제 해결 집중 필요
- 첫 고객의 모든 요청 수용: 제품 방향 왜곡
- 과도한 완성도 추구: 80% 완성으로 충분
- 팀 없이 혼자 결정: 고립된 PO는 편향 가중

## 실전 팁
- "만약 오늘 출시해야 한다면 무엇을 cut할 것인가?" 주기적 자문
- 첫 10명 고객을 직접 관리: 그들에게서 모든 것을 배운다
- 경쟁사보다 빠른 실험 속도 유지
- 주간 투자자 업데이트: 투명성이 신뢰를 만든다
- 번아웃 관리: 지속 가능한 페이스가 장기 생존의 조건
