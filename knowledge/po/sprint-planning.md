# 스프린트 계획 — 스프린트 계획과 실행
> 핵심 메시지: "스프린트는 학습 단위다. 기능 납품 단위가 아니다"

## 핵심 원칙
- 스프린트 목표(Sprint Goal)가 할 일 목록보다 중요하다
- 팀의 속도(Velocity)를 기반으로 현실적 계획 수립
- Definition of Done을 팀 전체가 합의해야 한다
- 스프린트 내 범위 변경은 원칙적으로 금지

## 판단 기준
- 이번 스프린트 목표를 한 문장으로 설명할 수 있는가?
- 선택한 항목들이 스프린트 목표에 기여하는가?
- 팀의 실제 가용 시간을 고려했는가? (휴가, 회의 등)
- 각 항목의 완료 기준이 명확한가?

## 실전 예시 (프레임워크)

### 스프린트 플래닝 아젠다 (2시간)
1. (15분) 스프린트 목표 합의: PO가 제안, 팀이 조정
2. (60분) 백로그 항목 선택: 우선순위 순으로 팀이 선택
3. (30분) 태스크 분해: 각 스토리를 구체적 태스크로
4. (15분) 확약: 팀이 스프린트 백로그에 합의

### 스프린트 목표 작성법
나쁜 예: "결제 기능 개발, 마이페이지 개선, 버그 수정"
좋은 예: "신규 사용자가 첫 결제를 완료할 수 있는 최소 플로우 완성"

### 스토리 포인트 추정 (플래닝 포커)
- 1점: 반나절 이하
- 2점: 하루 내외
- 3점: 2-3일
- 5점: 한 주
- 8점: 분해 필요
- 13점: 너무 큼, 반드시 분해

### 스프린트 용량 계산
- 팀 총 가용일 = (팀원 수 × 스프린트 일수) - 회의/휴가 일수
- Focus Factor 적용: 실제 개발 시간은 가용 시간의 70-80%
- 이전 3개 스프린트 평균 Velocity 참고

### 일일 스탠드업 (15분)
- 어제 한 것 (진행 공유)
- 오늘 할 것 (계획 공유)
- 블로커 (장애물 공유)
- 스프린트 번다운 차트 확인

### 스프린트 리뷰 구성 (1시간)
1. 스프린트 목표 달성 여부 발표
2. 완성된 항목 데모
3. 이해관계자 피드백 수집
4. 미완성 항목 처리 결정

### Definition of Done 체크리스트
- [ ] 기능 코드 완성
- [ ] 유닛 테스트 작성 및 통과
- [ ] 코드 리뷰 완료
- [ ] QA 테스트 통과
- [ ] 관련 문서 업데이트
- [ ] 프로덕션 배포 가능 상태

## 안티패턴
- 스프린트 목표 없이 할 일 목록만 있는 스프린트
- 스프린트 중반에 새 항목 추가 허용
- 팀 속도를 무시한 낙관적 계획
- "완료" 정의 없이 진행되는 개발
- 스탠드업이 상태 보고 회의로 변질

## 실전 팁
- 스프린트 첫날 팀 전체가 목표를 암기할 수 있어야 함
- 블로커는 24시간 내 해결 또는 에스컬레이션
- 스프린트 리뷰에 실제 사용자를 초청하면 피드백 품질 향상
- 번다운 차트를 매일 업데이트하여 팀이 진척을 시각적으로 확인
- 레트로스펙티브에서 1가지 개선 실험을 다음 스프린트에 적용
