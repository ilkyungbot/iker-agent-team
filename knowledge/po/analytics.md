# 데이터 분석 — 데이터 분석과 인사이트 도출
> 핵심 메시지: "데이터는 질문에 답하지만, 질문은 사람이 한다"

## 핵심 원칙
- Data-Informed, Not Data-Driven: 데이터는 도구, 판단은 사람이
- 상관관계 ≠ 인과관계: 항상 인과 메커니즘을 고민
- 좋은 질문이 좋은 분석을 만든다
- 분석 결과는 행동 가능한 인사이트로 변환되어야 한다

## 판단 기준
- 이 분석이 구체적인 의사결정에 영향을 주는가?
- 데이터의 출처와 신뢰도를 확인했는가?
- 세그먼트별로 다른 패턴이 있지 않은가?
- 상관관계를 인과관계로 잘못 해석하지 않는가?

## 실전 예시 (프레임워크)

### 분석 프로세스
1. 질문 정의: 무엇을 알고 싶은가?
2. 가설 수립: 예상하는 답은 무엇인가?
3. 데이터 수집: 어떤 데이터가 필요한가?
4. 분석 실행: 패턴과 이상치 탐색
5. 해석: 데이터가 말하는 것은 무엇인가?
6. 행동: 이 인사이트로 무엇을 할 것인가?

### 퍼널 분석 방법
```
단계 1: 방문 (100%)
    ↓ (70% 전환)
단계 2: 회원가입 (70%)
    ↓ (40% 전환)
단계 3: 첫 구매 (28%)
    ↓ (60% 전환)
단계 4: 재구매 (17%)
```
- 가장 큰 드롭오프 지점 식별
- 세그먼트별(디바이스, 유입 경로, 사용자 유형) 퍼널 비교

### 코호트 분석 템플릿
| 가입 월 | W1 | W2 | W4 | W8 | W12 |
|-------|----|----|----|----|-----|
| 1월 | 100% | 45% | 30% | 20% | 15% |
| 2월 | 100% | 50% | 35% | 22% | - |
- 리텐션 곡선의 평탄화(flattening) 지점이 핵심 사용자 기반

### 세그먼트 분석 기준
- 인구통계: 연령, 지역, 직업
- 행동: 활성 사용자 vs 비활성, 기능 사용 패턴
- 획득 경로: 오가닉, 유료, 추천
- 가치: 고가치 vs 저가치 사용자

### 이상치(Anomaly) 탐지
- 일별/주별 지표 변화율 모니터링
- Z-Score > 2 = 이상치 조사 필요
- 이상치 발견 시: 배포 이력, 마케팅 활동, 외부 이벤트 확인

### SQL 분석 패턴 (공통)
```sql
-- DAU 계산
SELECT date, COUNT(DISTINCT user_id) as dau
FROM events
WHERE event_name = 'session_start'
GROUP BY date

-- 리텐션 계산 (D7)
SELECT cohort_date,
  COUNT(DISTINCT u.user_id) as cohort_size,
  COUNT(DISTINCT CASE WHEN datediff(e.date, u.signup_date) = 7
    THEN e.user_id END) as d7_retained
FROM users u
LEFT JOIN events e ON u.user_id = e.user_id
GROUP BY cohort_date
```

## 안티패턴
- 가설 없는 탐색적 분석 → 결론 없는 리포트
- 확증 편향: 원하는 결론을 지지하는 데이터만 선택
- 샘플 크기 무시: 소수 데이터로 성급한 일반화
- 분석 마비: 더 많은 데이터를 기다리며 결정 지연
- 데이터 과신: 측정되지 않는 가치(질적 경험)를 무시

## 실전 팁
- 주요 지표 이상 시 5분 내 원인 파악할 수 있는 대시보드 구성
- 분석 노트북/문서를 팀이 공유하여 재사용 가능하게
- 정량 데이터와 정성 피드백을 함께 분석
- 분기별 데이터 품질 감사: 이벤트 누락, 중복, 오류 확인
- "So what?" 테스트: 인사이트가 행동으로 연결되는지 확인
