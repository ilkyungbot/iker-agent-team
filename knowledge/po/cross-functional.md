# 크로스 펑셔널 협업 — 크로스 펑셔널 팀 운영
> 핵심 메시지: "최고의 제품은 PO가 만드는 것이 아니라 팀이 함께 만든다"

## 핵심 원칙
- 역할별 전문성을 존중하되, 방향은 함께 결정한다
- 심리적 안전감이 없으면 최선의 아이디어가 나오지 않는다
- 투명한 정보 공유가 협업의 기반
- 팀의 성공을 개인의 성공보다 우선시

## 판단 기준
- 팀원들이 "왜"를 이해하고 작업하는가?
- 의사결정 권한이 명확하게 정의되어 있는가?
- 블로커가 즉시 공유되고 해결되는가?
- 팀이 성과를 함께 축하하는 문화가 있는가?

## 실전 예시 (프레임워크)

### PO-엔지니어 협업 원칙
- 요구사항이 아닌 문제를 공유: "이 기능을 만들어라"가 아닌 "이 문제를 해결해야 한다"
- 기술적 제약을 일찍 공유받기: 스프린트 중 서프라이즈 없도록
- 기술 부채 로드맵에 포함: 엔지니어의 장기 신뢰 획득
- "어떻게" 결정권은 엔지니어에게: PO는 "무엇"과 "왜"만

### PO-디자이너 협업 원칙
- 문제 정의 단계부터 디자이너 참여
- "이렇게 만들어 달라"가 아닌 "이 경험을 어떻게 설계할까?"
- 사용자 리서치를 함께 수행
- 디자인 결정에 사용자 데이터 연결

### PO-데이터 분석가 협업
- 분석 요청 전 명확한 질문과 의사결정 맥락 제공
- "데이터 뽑아 주세요" → "X 의사결정을 위해 Y를 알아야 합니다"
- 분석 결과 해석을 함께 논의
- 지표 정의와 측정 방법 사전 합의

### RACI 매트릭스
| 활동 | PO | 엔지니어 | 디자이너 | 분석가 |
|-----|-----|---------|---------|-------|
| 문제 정의 | R,A | C | C | C |
| PRD 작성 | R,A | C | C | I |
| 기술 설계 | I | R,A | I | I |
| UX 설계 | C | I | R,A | I |
| 지표 설계 | R | I | I | A,C |
| 스프린트 계획 | A | R | C | I |

R=Responsible, A=Accountable, C=Consulted, I=Informed

### 팀 협업 루틴
- 일일 스탠드업 (15분): 진행 상황, 블로커 공유
- 스프린트 플래닝 (2시간): 목표와 작업 합의
- 스프린트 리뷰 (1시간): 데모 및 피드백
- 레트로스펙티브 (1시간): 프로세스 개선
- 주간 싱크 (30분): PO-디자이너-엔지니어 리드 정렬

### 갈등 해결 프로세스
1. 직접 대화 먼저: 이메일/Slack이 아닌 1:1
2. 사실과 관찰 기반 대화: 추정이나 비난 금지
3. 공통 목표 확인: "우리 둘 다 원하는 것은..."
4. 옵션 탐색: "양쪽 모두 만족할 방법은?"
5. 에스컬레이션: 해결 불가 시 상위 관리자 개입

## 안티패턴
- Waterfall 사고: 디자인 완성 후 개발 시작
- 팀을 "자원"으로 보는 PO: 사람이 먼저
- 칭찬은 팀에게, 책임은 자신에게: 반대가 맞음
- 정보 사일로: 팀별 필요한 정보가 다른 팀에 숨겨짐
- 회의 과부하: 불필요한 회의로 개발 시간 낭비

## 실전 팁
- 팀원의 작업 맥락을 이해하기 위해 코드/디자인 툴 기본 학습
- 팀원 1:1 미팅을 격주로: 블로커와 성장 지원
- 팀의 성과를 경영진에게 적극적으로 홍보
- 팀 빌딩 활동을 분기별로: 신뢰 형성에 투자
- "팀을 위해 내가 어떤 장벽을 제거할 수 있는가?" 주간 질문
