# 백로그 관리 — 백로그 관리와 그루밍
> 핵심 메시지: "건강한 백로그는 팀의 방향을 명확히 하고, 불필요한 논의를 줄인다"

## 핵심 원칙
- 백로그는 우선순위가 매겨진 학습 목록이다
- 크기가 작을수록 좋다: 100개보다 30개의 잘 정의된 항목
- 정기적 그루밍(Refinement)으로 백로그를 항상 최신 상태로 유지
- 3개월 이상 된 항목은 삭제 또는 재검토

## 판단 기준
- 각 항목의 비즈니스 가치가 명확한가?
- 상위 10개 항목은 즉시 스프린트에 투입 가능한가?
- 백로그 항목들이 현재 전략과 정렬되어 있는가?
- 중복되거나 모순되는 항목은 없는가?

## 실전 예시 (프레임워크)

### 백로그 계층 구조
```
Epic (테마/목표, 분기 단위)
├── Story (사용자 스토리, 스프린트 단위)
│   ├── Task (기술적 태스크, 일 단위)
│   └── Task
└── Story
    └── Task
```

### 백로그 그루밍 주기
- 주간: 상위 10개 항목 정제 (30분)
- 격주: 전체 백로그 검토 (1시간)
- 분기: 전략 정렬 및 대규모 재우선순위화 (2시간)

### 그루밍 세션 아젠다
1. 신규 항목 추가 (리뷰 및 초기 우선순위)
2. 기존 항목 정제 (스토리 분해, 수락 기준 추가)
3. 우선순위 조정 (전략 변경 반영)
4. 오래된 항목 삭제 또는 보관
5. 상위 항목 추정 (스토리 포인트)

### 좋은 백로그 항목 기준 (INVEST)
- Independent: 다른 항목과 독립적으로 완료 가능
- Negotiable: 구현 방법이 유연함
- Valuable: 명확한 비즈니스/사용자 가치
- Estimable: 크기 추정 가능
- Small: 1-2 스프린트 내 완료 가능
- Testable: 완료 기준 검증 가능

### 백로그 항목 카드 템플릿
```
제목: [동사 + 목적어 형태]
유형: Story / Bug / Tech Debt / Research
우선순위: P0 / P1 / P2 / P3

사용자 스토리:
As a [사용자], I want to [행동], so that [이유]

수락 기준:
- [ ] 기준 1
- [ ] 기준 2

추정: [스토리 포인트]
관련 링크: [디자인/PRD/이슈]
```

### 백로그 항목 유형별 처리
| 유형 | 설명 | 비율 권장 |
|-----|------|---------|
| Feature | 신규 기능 | 60-70% |
| Bug | 버그 수정 | 10-15% |
| Tech Debt | 기술 부채 | 15-20% |
| Research | 조사/실험 | 5-10% |

## 안티패턴
- 수백 개의 항목이 쌓인 방치 백로그
- 그루밍 없이 스프린트 플래닝에서 처음 보는 항목들
- 삭제하지 못하는 심리적 백로그 집착
- 이해관계자 요청을 모두 백로그에 추가 ("No"가 없는 백로그)
- 오래된 항목의 맥락을 아무도 모르는 상황

## 실전 팁
- 백로그를 공개하여 이해관계자의 투명성 확보
- "이 항목이 3개월 내 스프린트에 들어갈 가능성이 있는가?" 기준으로 정리
- 삭제는 영구 소멸이 아님 → 아카이브에 보관하고 필요시 복구
- 유사 항목을 Epic으로 묶어 관리 단순화
- 스프린트 리뷰에서 미완료 항목 처리 결정을 팀이 함께
