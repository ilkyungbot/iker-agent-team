# 사용자 리서치 — 인터뷰, 설문, 유저 테스트
> 핵심 메시지: "사용자는 자신이 원하는 것을 알지 못하지만, 자신의 문제는 알고 있다"

## 핵심 원칙
- 문제를 발견하는 것이 목적, 솔루션 검증이 아님
- 듣기 80%, 말하기 20%: 인터뷰어가 많이 말하면 실패
- 과거 행동이 미래 의도보다 신뢰성 높다
- 정성 리서치와 정량 리서치는 상호보완적

## 판단 기준
- 이 리서치가 구체적인 의사결정에 도움이 되는가?
- 충분한 다양성을 가진 참여자를 선정했는가?
- 확증 편향을 방지하는 질문 설계인가?
- 행동(What they do)과 이유(Why they do)를 모두 파악하는가?

## 실전 예시 (프레임워크)

### 인터뷰 유형별 목적
| 유형 | 목적 | 소요 시간 |
|-----|------|---------|
| Problem Discovery | 문제 발굴 | 45-60분 |
| Solution Validation | 솔루션 검증 | 30-45분 |
| Usability Test | 사용성 평가 | 45-60분 |
| Continuous Discovery | 주간 학습 | 20-30분 |

### 인터뷰 질문 구성 (Problem Discovery)
**오프닝 (5분)**
- "오늘 인터뷰에 참여해 주셔서 감사합니다. 저는 [주제]에 대해 이해하고 싶습니다"
- "혹시 녹음해도 괜찮으시겠어요?"

**맥락 파악 (10분)**
- "현재 하시는 일을 간단히 설명해 주시겠어요?"
- "[주제 관련 활동]을 하실 때 전형적인 하루/주는 어떻게 보내시나요?"

**문제 탐색 (25분)**
- "가장 어려웠던 경험을 이야기해 주실 수 있나요?"
- "그때 어떻게 해결하셨나요?" (현재 대안 파악)
- "왜 그 방법을 선택하셨나요?"
- "그 과정에서 가장 불편한 점은 무엇이었나요?"

**마무리 (5분)**
- "마지막으로 제가 물어보지 않았는데 중요하다고 생각하시는 게 있나요?"

### Jobs To Be Done (JTBD) 프레임워크
- 기능적 Job: "나는 [구체적 행동]을 하고 싶다"
- 감정적 Job: "나는 [특정 감정]을 느끼고 싶다"
- 사회적 Job: "나는 [타인에게 어떻게 보이고] 싶다"

JTBD 인터뷰 핵심 질문:
"처음 이 서비스를 사용하게 된 계기가 무엇이었나요?"
"그 전에는 어떻게 하셨나요?"
"어떤 상황에서 이 서비스를 떠올리시나요?"

### 설문 설계 원칙
- 리커트 척도: 홀수(5점/7점) 사용
- 선도 질문 금지: "얼마나 유용했나요?" → "이 기능에 대해 어떻게 생각하세요?"
- 설문 길이: 5-10분 이내 (완료율 고려)
- NPS: "이 서비스를 지인에게 추천할 의향 (0-10)"

### 유저 테스트 프로토콜
1. 태스크 설명: "[목표]를 달성해 보세요" (방법은 알려주지 않음)
2. 생각 말하기(Think Aloud): "하시는 것을 말로 설명해 주세요"
3. 관찰: 행동과 말의 불일치에 주목
4. 사후 질문: "어떤 부분이 어려우셨나요?"

## 안티패턴
- 솔루션을 피칭하는 인터뷰: "이런 기능이 있다면 어때요?"
- 친구/가족 인터뷰: 객관성 부족
- 5명 이하로 결론 도출: 패턴 발견에 최소 5-8명 필요
- 인터뷰 후 즉시 결론 도출: 팀 함께 인사이트 분석 필요
- 리서치 결과를 실행하지 않음: 사용자를 실망시키는 패턴

## 실전 팁
- 매주 2명 인터뷰: 연속적 발견의 핵심 루틴
- 인터뷰 직후 5분 브리핑: 핵심 인사이트 즉시 기록
- 팀원을 인터뷰에 동석시켜 공감 형성
- 인터뷰 영상/녹음을 팀이 공유 가능하게 저장
- 인사이트 → 기회 → 솔루션 맵핑으로 체계화
