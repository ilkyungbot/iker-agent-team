# 케이스 스터디 — 성공/실패 사례 분석
> 핵심 메시지: "남의 실패에서 배우는 것이 내 실패에서 배우는 것보다 저렴하다"

## 핵심 원칙
- 성공 사례만큼 실패 사례에서 더 많이 배울 수 있다
- 맥락 없는 사례는 오히려 해롭다: 우리 상황에 맞는지 검토
- 원칙을 추출하되, 사례를 직접 복사하지 않는다
- 내부 사례를 외부 사례만큼 소중히 다룬다

## 핵심 원칙 (성공 패턴)
- 명확한 고객 문제 정의 → 솔루션 순서 준수
- 빠른 실험과 학습 → 시장 피드백 기반 반복
- 강력한 팀 문화 → 자율성과 책임감
- 데이터 기반 결정 + 고객 공감의 결합

## 판단 기준
- 이 사례의 핵심 원인이 무엇인가?
- 우리 상황과 어떤 점에서 유사하고 다른가?
- 이 교훈을 우리 제품에 적용할 수 있는가?
- 어떤 가정이 이 사례를 가능하게 했는가?

## 실전 예시 (성공 사례)

### Slack — B2B 바이럴 성장
**상황**: 게임 회사 내부 도구에서 시작
**전략**: 팀 단위 채택 → 회사 전체 확산 → 타 회사 확산
**핵심 교훈**:
- Aha Moment: 팀이 2,000개 메시지를 교환하는 순간
- Bottom-up 채택: 직원이 먼저 쓰고 IT 담당자가 나중에 승인
- 무료로 시작하여 가치를 증명 후 전환

### Airbnb — 크레드 없이 신뢰 구축
**상황**: 초기 숙박 공유 개념에 대한 거부감
**전략**: 직접 호스트 방문, 전문 사진 촬영
**핵심 교훈**:
- 스케일 안 되는 일을 먼저 하라 (Do things that don't scale)
- 신뢰는 기능이 아닌 경험으로 구축
- 공급(호스트) 품질이 수요보다 먼저

### Spotify — 데이터 기반 개인화
**상황**: 음악 스트리밍 레드오션 시장
**전략**: Discover Weekly 등 개인화 기능으로 차별화
**핵심 교훈**:
- 데이터 자산이 경쟁 우위
- 사용자 행동 데이터로 가치를 돌려주는 선순환
- 감성적 연결(음악)과 데이터 분석의 결합

## 실전 예시 (실패 사례)

### Google Glass — 시장 타이밍 실패
**실패 원인**:
- 사용자 문제보다 기술 중심 출발
- 프라이버시 우려 무시
- 가격 ($1,500) 과 가치의 불일치
**교훈**: 기술이 있어도 사용자 맥락을 무시하면 실패

### Quibi — 가설 검증 없는 $1.8B 투자
**실패 원인**:
- "통근 중 짧은 영상" 가설을 코로나로 검증 불가
- 출시 전 충분한 시장 검증 없음
- 스케일에 맞지 않는 콘텐츠 비용
**교훈**: 가장 위험한 가정을 먼저, 가장 저렴하게 검증하라

### Friendster — 기술 부채로 인한 몰락
**실패 원인**:
- 성장 속도를 기술 인프라가 따라가지 못함
- 사용자 불편을 기술 개선보다 기능 추가로 해결 시도
- 기술 부채를 미루다 결국 경쟁사에 기회 제공
**교훈**: 성장과 기술 부채 관리를 동시에

### 공통 실패 패턴
1. 솔루션 먼저, 문제는 나중
2. PMF 없이 스케일 시도
3. 핵심 지표 대신 Vanity Metrics 추적
4. 팀 문화 무시한 빠른 성장
5. 단일 채널 의존

## 안티패턴
- 성공 사례를 맥락 없이 직접 적용
- "우리는 그것과 다르다"는 근거 없는 예외 주장
- 실패 사례 분석을 부정적으로 보는 문화
- 내부 실패를 숨기고 외부 사례만 분석
- 교훈을 기록하지 않아 반복 실패
